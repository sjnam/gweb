\noindent
This is a guide to {\bf GWEB} for readers who already know {\bf CWEB}, or
literate programming generally. It does not re-explain what literate programming
is; it concentrates on how GWEB differs from CWEB and on the features and
commands you will actually use. To a first approximation GWEB is ``CWEB with C
replaced by Go,'' so most of your CWEB knowledge transfers intact --- the
interesting parts are where Go, and \cw{gofmt}, change the picture.

\chapter{GWEB at a glance}
GWEB mirrors CWEB tool for tool:

\bull \cw{gtangle} $\approx$ \cw{ctangle} --- extracts the compilable Go source;

\bull \cw{gweave} $\approx$ \cw{cweave} --- typesets the documentation;

\bull \cw{gwebmac.tex} $\approx$ \cw{cwebmac.tex} --- the \TeX\ macros the woven
output uses.

\noindent A web is a single file, conventionally \cw{foo.w}. \cw{gtangle foo}
writes \cw{foo.go} (plus any extra files); \cw{gweave foo} writes \cw{foo.tex},
which \cw{pdftex} turns into \cw{foo.pdf}. As in CWEB, the \cw{.w} may be left
off the command line.

\chapter{What carries over from CWEB}
If you know CWEB you already know most of GWEB. A web is a sequence of numbered
{\it sections}, each with up to three parts in order --- a \TeX\ commentary
part, a definition part (\cw{@d}/\cw{@f}/\cw{@s}), and a code part --- preceded
by {\it limbo}. The control codes keep their familiar meanings:

\bull \cw{@*}, \cw{@*1}, \cw{@*2}, $\ldots$ start starred (chapter) sections;
\cw{@\ } starts an ordinary section.

\bull \cw{@c} starts unnamed code; \cw{@<Name@>=} defines a refinement and
\cw{@<Name@>} references one. A trailing \cw{...} abbreviates a name to its
unique prefix.

\bull \cw{@(file@>=} sends the following code to another output file; \cw{@i}
includes another web.

\bull \cw{@f}/\cw{@s} format an identifier; \cw{@\char94 text@>}, \cw{@.text@>},
\cw{@:text@>} add index entries; \cw{@!} forces an index definition.

\bull \cw{@=...@>} is verbatim, \cw{@t...@>} is \TeX\ text, \cw{@\char38} pastes
tokens, \cw{@@} is a literal \cw{@}, and \cw{@\char37} and \cw{@q...@>} are
comments.

\bull \cw{@,} \cw{@/} \cw{@|} \cw{@\#} fine-tune the woven layout.

\bull change files (\cw{@x}/\cw{@y}/\cw{@z}) and inline code \cw{|...|} in
commentary work exactly as in CWEB.

\noindent Two small conveniences: \cw{@p} is accepted as a synonym for \cw{@c}
(as in CWEB), and you never write \cw{\bs input gwebmac} --- \cw{gweave} emits it
for you, and removes a stray copy if you include one.

\chapter{What makes GWEB different}
The differences all follow from typesetting Go instead of C.

\section{No preprocessor}
Go has no preprocessor, so \cw{@d} never generates code: \cw{gtangle} ignores it
(use Go \cw{const} and \cw{func} for real definitions). It is not useless,
though --- \cw{gweave} reads \cw{@d Name} as a request to set \cw{Name} in
typewriter, like a CWEB macro (see below). The formatting directives \cw{@f} and
\cw{@s} also remain --- for example \cw{@f Stack int} sets the user type
\cw{Stack} in bold, the way predeclared types are set.

\section{Pretty-printing mirrors gofmt, not a grammar}
This is the central design choice, and the largest departure from \cw{cweave}.
\cw{cweave} parses C and re-derives every space and break from the grammar.
\cw{gweave} does not parse Go at all: it reproduces the spacing that \cw{gofmt}
has already imposed. Because gofmt-formatted Go encodes the grammar {\it in\/}
its spacing, this gets the hard cases right for free --- the pointer \cw{*T}
versus the product \cw{a * b}, the slice \cw{[]T} versus the index \cw{a[i]},
precedence spacing such as \cw{a*b + c} --- with no parser to maintain and no
trouble with generics or future syntax. The practical rule is simply: {\it write
the code in your sections the way \cw{gofmt} would.}

A consequence is that CWEB's prettyprinter hints \cw{@+}, \cw{@[}\thinspace
$\ldots$\thinspace\cw{@]}, and \cw{@;} have nothing to do, and are accepted and
ignored. The explicit layout codes \cw{@,} \cw{@/} \cw{@|} \cw{@\#} still work
for the occasional manual touch-up.

\section{gtangle runs gofmt}
The tangled output is passed through \cw{gofmt}, so a correct web always yields
canonically formatted Go. If the assembled text is not valid Go, \cw{gtangle}
keeps the unformatted output and prints a warning instead of failing.

\section{Types and constants are classified automatically}
\cw{gweave} scans for \cw{type} and \cw{const} declarations --- both the single
form (\cw{type Name ...}, \cw{const Name ...}) and the parenthesized block form
--- and gives each declared name a CWEB-style class wherever it appears: a
\cw{type} is set {\bf bold}, like a predeclared type, and a \cw{const} is set in
typewriter, the way \cw{cweave} sets a name defined by \cw{@d} (a Go \cw{const}
is the closest analogue). For names you do {\it not\/} declare in this web ---
another package's constants, say --- mark them yourself with \cw{@d}: \cw{@d
http.StatusOK} sets \cw{StatusOK} in typewriter everywhere it appears. You can
override any automatic class with \cw{@f}/\cw{@s} --- e.g.\ \cw{@f Name int}
forces \cw{Name} bold, or \cw{@f Name x} (any ordinary identifier) restores plain
italic.

\section{GWEB is written in itself}
Like CWEB, GWEB is a literate program: each tool is tangled from a \cw{.w} web
kept beside its Go. The committed Go is byte-for-byte reproducible from those
sources --- a fixpoint the test suite checks on every change --- so the program
you read and the program that runs cannot drift apart.

\chapter{The woven document}
\cw{gweave} produces the same furniture as \cw{cweave}: run-in bold section
headings (with a page break before each major starred section), a two-column
index, a {\it Names of the Sections\/} list, a contents page, and section-number
running heads. A whole section --- its commentary and its code --- is kept
together on one page when it fits, and a code part is never split across a page
boundary unless it is itself taller than a page.

\section{How code is set}
Reserved words and user-defined types are bold, other identifiers italic, string
and rune literals typewriter (in \cw{cmtex10}, as in CWEB), numbers and comments
roman. A few Go-specific touches: the shift operators \cw{<<} and \cw{>>} are set
as the tight symbols $\ll$ and $\gg$; a thin space follows a function name or a
keyword before \cw{(}, so \cw{f (x)} does not jam together; and a comment's
\cw{//} is kerned a little tighter. \TeX\ mathematics written between dollar
signs inside a comment is typeset as mathematics.

\section{Index and formatting codes}
The hand-index codes \cw{@\char94 text@>} (roman), \cw{@.text@>} (typewriter),
and \cw{@:text@>} (formatted by \cw{\bs9}) work as in CWEB, as do \cw{@!} to
force a definition entry and \cw{@f}/\cw{@s} to reclassify a name. An identifier
is marked as ``defined'' (its index number underlined) when it follows \cw{func},
\cw{var}, \cw{const}, or \cw{type}, or sits just left of \cw{:=}; this is a
heuristic, not a type analysis, and \cw{@!} overrides it.

\section{Hyperlinks and bookmarks}
Section numbers shown as references, in the index, in cross-reference notes, and
on the contents page become links, and the starred sections form a PDF outline.
Two back ends produce them, exactly as in CWEB:

\bull with \cw{pdftex} in PDF mode, the engine's own primitives;

\bull through DVI, by requesting the \cw{\bs special} back end with
\cw{\bs let\bs pdf+} and converting the result with \cw{dvipdfmx}.

\noindent A plain DVI run with no \cw{\bs pdf} simply omits the links and outline.

\section{Helper macros for commentary}
\cw{gwebmac.tex} carries the cwebmac niceties for prose: \cw{\bs.\char123...\char125}
sets typewriter text (with \cw{\bs\bs}, \cw{\bs\char123}, $\ldots$ for literal
specials), \cw{\bs\bs\char123 id\char125} sets an italic identifier, and the
language tags \cw{\bs CEE/}, \cw{\bs GO/}, \cw{\bs UNIX/}, \cw{\bs TEX/} are
available. \cw{\bs,} is a thin space in math or text. The GWEB idiom for inline
code, though, remains simply \cw{|...|}.

\section{Customizing the page}
Redefine these in limbo (after the auto-emitted \cw{\bs input gwebmac}):

\bull \cw{\bs def\bs title\char123...\char125} --- the running-head and contents
title (it defaults to the upper-cased file name, and is shown all-caps in the
header);

\bull \cw{\bs pagewidth}, \cw{\bs pageheight} --- the page size;

\bull \cw{\bs def\bs contentspagenumber\char123 n\char125} together with
\cw{\bs pageno} --- to renumber for binding into a larger document;

\bull \cw{\bs Gcodepenalty} (how strongly a code part resists a page break),
\cw{\bs Gcommentkern} (the \cw{//} tightening), and \cw{\bs Gthinmuskip} (the
thin space before \cw{(}, \cw{2mu} by default) are registers you may tune.

\chapter{Running the programs}
\begincode
gtangle [-o dir] file[.w] [change[.ch]]
gweave  [-o dir] file[.w] [change[.ch]]
|endcode
The \cw{.w} extension --- and a change file's \cw{.ch} --- may be omitted.
\cw{-o} chooses the output directory. \cw{-version} prints the version and exits.
As each tool runs it prints a one-line banner and then a \cw{*}{\it n\/} progress
marker for every starred section, as CWEB's tools do.

\cw{gtangle} always emits \cw{//line} directives in the tangled Go --- as CWEB's
\cw{ctangle} always emits \cw{\#line} --- so the compiler, \cw{go vet}, and panic
traces report errors at \cw{.w} positions rather than \cw{.go} ones. GWEB tangles
its own sources this way too, so editing a \cw{.w} reshuffles those line numbers
in the committed Go.

To make a PDF, either go straight through \cw{pdftex}, or through DVI and
\cw{dvipdfmx} for the same hyperlinks:
\begincode
pdftex foo.tex                       # native PDF, with links
tex "\let\pdf+ \input foo.tex"       # or via DVI...
dvipdfmx foo.dvi                     # ...then to PDF, with links
|endcode
Make sure \TeX\ can find \cw{gwebmac.tex} (the installer below puts it on the
search path; otherwise set \cw{TEXINPUTS}).

Both tools report, each with a \cw{file:line} location, an unterminated control
code, a reference to an undefined section, an ambiguous \cw{...} abbreviation, or
a section defined but never used. These are warnings; \cw{gtangle} stops only if
it must actually expand a reference that was never defined.

\chapter{Installing GWEB}
For the commands alone, the Go way:
\begincode
go install github.com/sjnam/gweb/cmd/gtangle@latest
go install github.com/sjnam/gweb/cmd/gweave@latest
|endcode
For the full installation --- the commands, \cw{gwebmac.tex} placed in your \TeX\
tree so \cw{pdftex foo.tex} just works, and the man pages --- run
\cw{./install.sh} from a checkout (\cw{./install.sh --uninstall} removes it). The
repository also includes a VS\thinspace Code syntax highlighter for \cw{.w} files
under \cw{editors/vscode}, and this manual is built with \cw{make manual}.

\chapter{The bundled examples}
The \cw{examples/} directory holds five worked webs, from simple to rich:

\bull \cw{wc.w} --- a word counter; shows \cw{@f} and a change file (\cw{wc.ch});

\bull \cw{pmap.w} --- a generic concurrent map (goroutines, channels, a closure);

\bull \cw{seq.w} --- composable lazy iterators (generics, range-over-func);

\bull \cw{floyd.w} --- Floyd's partition problem, after Knuth's essay {\it Are
Toy Problems Useful?};

\bull \cw{sham.w} --- a port of Knuth's Stanford GraphBase demonstration program
\cw{sham}, built on the \cw{go-sgb} module.

\noindent GWEB also documents itself: \cw{make selfdoc} weaves the system's own
\cw{.w} sources into a single PDF.
